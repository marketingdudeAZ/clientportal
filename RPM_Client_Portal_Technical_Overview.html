RPM Living Client Portal — Technical Overview

Last updated: April 2026  ·  Prepared for RPM Living marketing team

What It Is

The RPM Client Portal is a branded self-service hub for RPM Living property clients, hosted on HubSpot CMS at go.rpmliving.com. It gives clients visibility into their marketing performance, active campaigns, spend details, and support tickets — all connected live to HubSpot CRM, HubSpot Service Hub, and the broader RPM marketing stack.

The portal is a single HubSpot page template (demo.html) that talks to a Flask API server (the "webhook server") hosted on Railway at https://clientportal-production-05ac.up.railway.app. The Flask server is the only thing that holds API keys and communicates with HubSpot.

Architecture at a Glance

Client Browser (go.rpmliving.com) │ │ fetch() calls over HTTPS ▼ Railway Flask API Server (clientportal-production-05ac.up.railway.app) │ ├── HubSpot CRM v3/v4 API (company, deal, ticket, note, contact data) ├── HubSpot Conversations API (ticket threads + replies) ├── HubSpot CMS Site Search API (knowledge base search) ├── HubSpot HubDB (assets, recommendations, budget tiers) ├── Google BigQuery (portfolio data, red-light signals) └── Anthropic Claude API (digest summaries, KB article drafts)

The portal page is a static HTML/JS file deployed to HubSpot's Design Manager. There is no separate database — all persistent data lives in HubSpot CRM or HubDB.

Portal Sections

1

Dashboard / Overview

Shows the client's property name, account manager, and a high-level health summary. Data comes from /api/property which reads HubSpot company properties. The AM who owns the company record in HubSpot is auto-surfaced as the client's primary contact.

2

Performance

Embeds reporting data wired to NinjaCat (configurable via NINJACAT_DASHBOARD_ID env var). Gauges and score cards are rendered from the /api/property response which includes channel-level performance metrics stored on the HubSpot company record.

3

Recommendations Feed

Pulls from HubDB table (HUBDB_RECOMMENDATIONS_TABLE_ID) — a structured table of recommendations the AM has entered for this client. Clients can approve or dismiss each recommendation directly from the portal:

  • Approve → POST /api/approve — logs to HubSpot, creates a ClickUp task in the appropriate service list, optionally triggers Claude to generate a campaign brief
  • Dismiss → POST /api/dismiss — logs dismissal to HubSpot
4

Digest

A Claude-generated plain-English summary of what's happening with the client's account. Calls /api/digest which reads recent HubSpot activity, sends a structured prompt to Claude Sonnet, and caches the result for 24 hours.

5

Asset Library

Displays marketing assets (photos, videos, brand files) stored in HubDB (HUBDB_ASSET_TABLE_ID). Clients can browse by category, open a lightbox viewer, and upload new files via POST /api/asset-upload which stores them in HubSpot File Manager and writes metadata to HubDB.

6

Included Services

Renders a pricing breakdown of the client's included services based on their unit count and tier selections. Pricing tiers are defined in config.py (CRM Lead Management, Website Hosting, SOCi, Training bands, etc.).

7

Budget Configurator

An interactive package builder. Clients adjust their SEO tier, Paid Search, Paid Social, Reputation, and add-ons — a running total updates live. On submit, POST /api/configurator-submit creates or updates a HubSpot deal and notifies the AM.

8

Support Tickets

Creating a Ticket

When a client submits a ticket, the portal first runs KB deflection:

  1. Calls GET /api/kb-search?q=<title> — searches HubSpot's Knowledge Base for matching articles
  2. If articles are found, shows them inline: "We found these articles that may help — did this answer your question?"
  3. Client clicks Yes (form closes) or No, I still need help (proceeds to submit)
  4. If no articles found, submits immediately without the deflection panel

On submission, POST /api/ticket creates the ticket in HubSpot's Support Pipeline (ID: 0), auto-assigns it to the AM who owns the company record, and associates it with both the company and contact records.

Conversation Threads

Each ticket card has a View Thread button. On click:

  1. Calls GET /api/ticket/<id>/thread — queries HubSpot Conversations API for the message thread
  2. Renders messages inline (incoming = client, outgoing = AM/RPM team)
  3. Shows a reply bar — client types a message and hits Enter or Send
  4. Reply is posted via POST /api/ticket/<id>/reply → writes back to HubSpot Conversations
  5. The AM sees the reply immediately in HubSpot Service Hub

Auto KB Draft on Close

When a ticket is moved to Closed, a background thread automatically:

  1. Fetches the full ticket subject, description, and conversation thread
  2. Calls Claude Haiku (claude-haiku-4-5-20251001) to write a Knowledge Base article draft
  3. Posts the draft as a HubSpot Note on the ticket tagged [KB DRAFT — Ready to publish]
  4. The AM copies it into the HubSpot KB UI (a ~30-second task)
HubSpot has no public API for creating KB articles. This Note-based workaround keeps the workflow intact without any manual API workarounds.
9

Spend Sheet

A full table of marketing spend across all RPM-managed properties. The data pipeline runs on first load, is cached for 30 minutes, and is pre-warmed on server startup:

  1. CRM search: all companies where plestatus is RPM Managed, Onboarding, or Dispositioning
  2. Batch associations: companies → deals (HubSpot v4 API, returns 207 Multi-Status)
  3. Batch read: latest deal per company
  4. Batch associations: deals → line items
  5. Batch read: line items (Search, PMax, Paid Social, SEO, Management Fee)
  6. Batch associations: deals → quotes
  7. Batch read: latest quote per deal (name + approval status)

Three filter inputs (Property Name, RPM Market, Marketing Manager) filter the table client-side.

10

Client Brief

GET /api/client-brief returns a structured summary of the client's onboarding status, goals, and key context. AMs update fields via PATCH /api/client-brief, which writes back to HubSpot company properties.

11

Portfolio (Red Light)

GET /api/portfolio pulls portfolio-level data from BigQuery. The Red Light system ingests performance signals and flags underperforming properties for AM attention.

Authentication & Security

  • All API calls include an X-Portal-Email header injected by HubSpot personalization tokens — the server uses this to identify the requesting client
  • API keys (HubSpot, Anthropic, ClickUp, Google) are stored only as Railway environment variables — never in code
  • CORS is locked to the HubSpot portal domain
  • HubSpot webhook payloads are validated via HMAC signature (hmac_validator.py)

Known Quirks & Notes

  • HubSpot v4 Batch Associations returns 207 Multi-Status (not 200). All batch association calls check for both status codes — this was the root cause of the spend sheet taking 2+ minutes to build before the fix.
  • KB article creation: HubSpot has no public API to create KB articles. The workaround is posting Claude's draft as a HubSpot Note tagged [KB DRAFT] on the resolved ticket.
  • Spend sheet first load: ~60 seconds on a cold server (1,300+ companies × multiple batch API calls). A background thread pre-warms the cache on startup; subsequent loads are under 1 second.
  • HubSpot CSP: HubSpot injects upgrade-insecure-requests on hosted pages, which upgrades http:// fetch calls to https://. This is why the API server must be on Railway with a real TLS certificate rather than localhost.
